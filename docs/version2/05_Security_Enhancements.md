# QuickChat Version 2: Security & Cryptography Enhancements

## 1. Executive Summary

QuickChat V1 establishes a strong security baseline using X25519, Ed25519, and the Noise Protocol (XX handshake). For V2, the security posture must evolve to address the complexities of a plugin ecosystem, enterprise compliance requirements, multi-device synchronization, and future cryptographic threats.

## 2. Multi-Device Synchronization & Identity

### 2.1. The Limitation of V1
V1 assumes a 1:1 mapping between a user and a single device (single key pair). Modern communication requires users to access their identity across laptops, workstations, and mobile devices.

### 2.2. V2 Solution: Device Provisioning
- **Primary Device:** The first device generated acts as the "Root" identity.
- **Provisioning Flow:** To add a new device, the user scans a QR code (or enters a short code) generated by the Primary Device. The devices perform a mutually authenticated Noise handshake.
- **Key Delegation:** The Primary Device signs a certificate delegating authority to the new device's subkey.
- **Message Routing:** Contacts encrypt messages using the Root public key, which are then fanned out (via Relay or direct P2P mesh) and encrypted for all known, valid device subkeys.

## 3. Plugin Security & Sandboxing

As defined in the Plugin Ecosystem spec, the WASM runtime provides the primary boundary.
- **Cryptographic Isolation:** Plugins *never* have direct access to the user's private keys.
- **Host-Mediated Encryption:** If a plugin needs to send a secure message, it passes the plaintext to the host via the `host_send_message` API. The host performs the ChaCha20-Poly1305 encryption.
- **Verified Plugins:** Introduction of cryptographic signatures for plugins. The QuickChat client will verify the signature of a plugin against a known registry or Enterprise CA before loading it, preventing supply-chain attacks.

## 4. Enterprise Audit & Compliance

Enterprise environments require visibility without compromising End-to-End Encryption (E2EE).
- **Tamper-Evident Audit Logs:** QuickChat Enterprise instances will log metadata (connection times, IP addresses, file transfer sizes, plugin executions) to a local, append-only cryptographic ledger (e.g., using a Merkle tree structure).
- **Data Loss Prevention (DLP) (Opt-in):** In strict enterprise deployments, an organization may require an "Escrow Key" to be added to group chats to scan for exfiltration. This must be explicitly transparent to all users in the chat via persistent UI warnings.

## 5. Forward-Looking Cryptography: Post-Quantum (PQC)

The threat of "Store Now, Decrypt Later" (SNDL) by quantum computers necessitates proactive upgrades.
- **Hybrid Key Exchange:** In V2/V3, QuickChat will transition the Noise handshake to a hybrid model. It will combine the existing X25519 ECDH with a NIST-standardized Post-Quantum KEM (e.g., Kyber/ML-KEM).
- **Rationale:** If the PQC algorithm turns out to have a flaw, the classical X25519 exchange still protects the data. This provides defense-in-depth against both current and future threats.

## 6. Secure Storage Upgrade

V1 uses basic file encryption for local keys. V2 will integrate deeply with OS-level secure enclaves.
- **macOS:** Secure Enclave / Keychain API.
- **Windows:** TPM / Credential Guard.
- **Linux:** Secret Service API / Keyring.
This prevents private key extraction even if the host filesystem is compromised by malware.
